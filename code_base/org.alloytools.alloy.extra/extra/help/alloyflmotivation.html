Motivation for AlloyFL

Writing correct declarative models that represent non-trivial properties is not easy, especially for practitioners who are not well-versed with the intricate syntax and semantics of declarative languages. Case in point, Daniel Jackson, the inventor of Alloy, has pointed out in his ICSE keynote that declarative specifications can be "maddening harder to learn and even harder to debug" and the "unsat coreis not enough". This motivates us to develop fault localization techniques for declarative models written in Alloy.

AlloyFL looks to adapt fault localization techniques for imperative languages (Java, C, C++) to Alloy's declarative execution environment. In particular, for an Alloy execution, all executed constraints are considered at the same time and produce any number of instances as output. Furthermore, a single line in an Alloy model can represent a complex specification, compared to a traditional line of code. To account for these changes, (1) we adapt a combination of traditional coverage-based fault localization and mutation-based fault localization techniques and (2) we localize faults to abstract syntax tree (AST) nodes rather than lines of code.
